# Deployment

Faro should use Vercel as the public app host for now.

The current app is a Next.js project with repo-versioned public data plus
private production services for authenticated workflows. Vercel serves the app,
Clerk handles authentication, Neon Postgres stores structured private product
state, and Cloudflare R2-compatible storage holds private Aportes attachments.
Operational checklists live in
[`docs/operations/production-runbook.md`](operations/production-runbook.md).

## Production Shape

- Public app: Vercel, currently `faro-ilumina.vercel.app`.
- Data artifacts: generated during `npm run build`.
- Heavy exports: static files under `public/exports/`, generated by `npm run exports:build` before `next build`.
- Auth: Clerk for private investigation folders and internal review surfaces.
- Product database: Neon Postgres via `DATABASE_URL`.
- Private storage: Cloudflare R2-compatible bucket for Aportes payloads and
  attachments.
- Hetzner VPS: reserved for ETL, cron jobs, Postgres/PostGIS, or heavier data
  services only if Vercel/Neon/R2 stop covering the operational need cleanly.

The public corpus is still generated from checked-in official snapshots. Private
Carpetas and Aportes review state are runtime product state and require the
production services above.

## Environment Checklist

Set these in Vercel Production before treating the deployment as production:

- `NEXT_PUBLIC_SITE_URL`
- `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`
- `CLERK_SECRET_KEY`
- `NEXT_PUBLIC_CLERK_SIGN_IN_URL`
- `NEXT_PUBLIC_CLERK_SIGN_UP_URL`
- `NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL`
- `NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL`
- `DATABASE_URL`
- `FARO_ADMIN_EMAILS`
- `FARO_REVIEWER_EMAILS`
- `STORAGE_ENDPOINT`
- `STORAGE_BUCKET`
- `STORAGE_ACCESS_KEY`
- `STORAGE_SECRET_KEY`

`NEXT_PUBLIC_ARCGIS_API_KEY` is optional and only raises Esri Wayback quota.
Do not set `FARO_ENABLE_TEST_AUTH` in production.

## Clerk Roles

Faro resolves roles from Clerk metadata first, then from explicit env allowlists:

- `role: "admin"` in Clerk metadata or email in `FARO_ADMIN_EMAILS`;
- `role: "reviewer"` in Clerk metadata or email in `FARO_REVIEWER_EMAILS`;
- everyone else defaults to `investigator`.

Admin/reviewer setup must be verified with real Clerk users. Do not document a
reviewer or admin as configured until the user can sign in and reach only the
surfaces intended for that role.

## Neon Migrations

Product migrations live in `data/product/migrations/*.sql` and are applied with:

```bash
npm run db:migrate
```

Before applying migrations, confirm `DATABASE_URL` points to the intended Neon
database or branch. For release work, apply against a preview/staging branch
first when available, then production. The current migration runner applies SQL
files in lexicographic order and expects migrations to be idempotent.

## R2 Storage Verification

Aportes attachments require private R2-compatible storage. Configure
`STORAGE_ENDPOINT`, `STORAGE_BUCKET`, `STORAGE_ACCESS_KEY` and
`STORAGE_SECRET_KEY`, then run:

```bash
npm run verify:aportes-storage
```

The smoke check writes a technical test aporte and verifies that both the
attachment object and manifest object exist in the bucket. Passing this check
does not make any Aporte public; it only verifies private storage wiring.

## Vercel Function Limits

Do not return full data collections from Vercel Functions. Vercel Functions have a hard response payload limit, so collection endpoints must stay paginated or redirect to static artifacts.

Current API contract:

- `/api/cases` returns paginated case collections, default `limit=100`, max `limit=200`.
- `/api/signals` returns paginated signal rows, default `limit=100`, max `limit=200`.
- `/api/export/[id]` returns one case evidence pack and can remain a Function response.
- `/api/export` redirects core exports to generated static files when available.
- `/api/export` returns a bounded manifest for ad hoc large filtered exports that do not have a static artifact.

## Build Commands

```bash
npm run data:verify
npm run data:geo-report
npm run data:quality-report
npm test
npm run typecheck
npm run build
```

`npm run build` runs `npm run exports:build` first, so Vercel deploys include the generated static export artifacts without committing those large generated files.

For production releases that touch private persistence or Aportes storage, also
run:

```bash
npm run db:migrate
npm run verify:aportes-storage
```

For documentation-only changes, `git diff --check` is sufficient unless the
change modifies code, config, generated data or release scripts.

## Hetzner Boundary

Use the Hetzner skill before touching the VPS. Administrative SSH should go through Tailscale only. Do not move the public UI to Hetzner unless there is a concrete reason such as long-running jobs, a database service, private object storage, or operational constraints that Vercel cannot cover cleanly.
